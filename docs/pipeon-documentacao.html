01 Visao Geral da Arquitetura

O Pipeon e um sistema full-stack com frontend React e dois backends alternativos: Express local para desenvolvimento e Cloudflare Worker para producao.

Regra critica de banco Procedimentos so podem operar contra o banco target-database. Nunca apontar logica de procedimento para outros bancos.

02 Fluxo de Autenticacao

Login baseado em e-mail sem senha visivel. O JWT retornado carrega identidade e role do usuario.

  1. Login — rota / Usuario informa o e-mail. Frontend chama POST /api/auth/login. Servidor verifica em pipeon_users e retorna um JWT assinado.
  2. Selecao de sistema — rota /select-system Lista os SystemDefinition salvos no localStorage. Usuario escolhe qual sistema deseja acessar.
  3. Selecao de sessao — rota /select-session Exibe as sessions do sistema escolhido (ex: target-database, pipeon). Cada session define o banco-alvo que sera usado nas operacoes.
  4. Menu principal — rota /menu Dashboard com acoes filtradas pela role e permissoes do usuario. Token e config da sessao ficam no sessionStorage.

Payload do JWT

{
  "sub":   "65a1b2c3d4e5f6a7b8c9d0e1",  // MongoDB _id do usuario
  "email": "operador@example.com",
  "name":  "Nome Completo",
  "role":  "operator",             // admin | operator | user
  "exp":   1748980800             // 8 horas a partir do login
}
Sessao expirada Quando o token expira (401), a lib atlas.ts limpa o sessionStorage automaticamente e redireciona para / com mensagem de erro.

/procedure/ — Lista de Procedimentos

Exibe todos os procedimentos disponiveis para execucao. Valida que o banco ativo e target-database antes de renderizar.

  • Entrada hardcoded: "Retornar avaliacoes para pendentes para substituicao de avaliador(a)" — direciona para /procedure/reset-evaluations
  • Entradas dinamicas: buscadas via GET /api/pipeon/procedures, filtrando isActive: true — cada uma direciona para /procedure/$id

/procedure/reset-evaluations — Reset de Avaliacoes

Procedimento especifico com logica propria. Fluxo em 4 etapas com stepper visual:

  1. IdentificacaoNumero do chamado (ticket) + ObjectId do ciclo (notice)
  2. Selecao do avaliadorBusca avaliadores via aggregate na colecao evaluations do banco target-database
  3. ConfirmacaoDry-run mostrando status atuais e campos que serao removidos. Opcao de agendar para depois.
  4. ExecucaoBackup ZIP local + backup no banco pipeon + reset de status + limpeza de campos phaseForm

Operacoes executadas no banco target-database:

// Passo 1 — Backup local (ZIP) e remoto (pipeon_auto_backups)

// Passo 2 — Reset de status
updateMany("evaluations", { filter }, {
  $set: { status: "NOTSTARTED" },
  $unset: {
    evaluationAverage: "", evaluationAverageByBlocks: "",
    evaluationSum: "", submittedDate: "", evaluatorNote: ""
  }
})

// Passo 3 — Limpeza do formulario
updateMany("evaluations", { filter }, {
  $unset: {
    "phaseForm.blocks.$[].fields.$[].value": "",
    "phaseForm.blocks.$[].sumValue": ""
  }
})

/procedure/$id — Executor Dinamico

Executa qualquer procedimento registrado em pipeon_procedures. Parametro id e o _id do documento no MongoDB.

  • Busca a definicao do procedimento via GET /api/pipeon/procedures
  • Renderiza formulario com base em procedure.inputs
  • Executa cada step em sequencia, interpolando variaveis {{nome}}
  • Registra log em pipeon_operations ao final

/admin — Painel de Administracao

Exclusivo para role admin. Organizado em 4 abas:

Aba Usuarios
  • Lista todos os usuarios de pipeon_users
  • Criar usuario: POST /api/users com campos name, email, password, role
  • Editar role ou status ativo: POST /api/users/:id/update
Aba Permissoes
  • Configura quais permissoes os roles operator e user possuem
  • Permissoes disponíveis: procedure, history, restore
  • Role admin sempre tem acesso total — nao e configuravel
  • Persiste no documento _id: "role_permissions" em pipeon_settings
  • Cache local em localStorage, sincronizado no login
Aba Sistemas

Gerencia a lista de SystemDefinition[] salva em localStorage (chave: pipeon-systems). Cada sistema agrupa uma ou mais sessions (conexoes banco/ambiente).

Estrutura SystemDefinition

{
  "id":               "target-prod",          // identificador unico (string)
  "name":             "[INTERNAL-SYSTEM] Expresso",
  "description":      "Ambiente de producao", // opcional
  "connectionString": "mongodb+srv://...",    // opcional — se vazio usa localhost
  "createdAt":        "2026-06-03T00:00:00.000Z",
  "sessions": [
    {
      "id":          "sample-prod-conn",
      "label":       "target-database",         // nome exibido ao usuario
      "database":    "target-database",         // nome exato do banco no MongoDB
      "environment": "production"            // "production" | "staging" | "local"
    }
  ]
}

Sistema padrao (seed automatico)

{
  "id":   "local",
  "name": "Banco Local",
  "description": "MongoDB local — mongodb://localhost:27017/",
  "connectionString": undefined,  // usa localhost por padrao
  "sessions": [
    { "id": "sample-local-conn", "label": "target-database", "database": "target-database", "environment": "local" },
    { "id": "local-pipeon",        "label": "pipeon",        "database": "pipeon",        "environment": "local" }
  ]
}

Como funciona no fluxo de login:

  • Em /select-system: o usuario escolhe um SystemDefinition. O ID escolhido fica em sessionStorage (chave: pipeon-selected-system)
  • Em /select-session: o usuario escolhe uma SystemSession do sistema. O database e a connectionString do sistema pai sao salvos no AtlasConfig em sessionStorage
  • Todas as chamadas atlasCall() usam o database e connectionString dessa config ativa
Aba Procedimentos
  • Wizard de 3 etapas para registrar procedimentos via IA
  • Etapa 1: Preenche manualmente nome, descricao, inputs e steps do procedimento
  • Etapa 2: Preview do procedimento — nome, descricao, tabela de inputs, cards de steps
  • Etapa 3: Selecao do projeto — o procedimento herda o projectId do projeto escolhido
  • Lista procedimentos cadastrados com opcao de excluir

/scheduled — Agendamentos

Lista todos os procedimentos agendados. Acoes disponiveis por item:

  • Executar agora: redireciona para a rota do procedimento com dados pre-preenchidos via sessionStorage (chave: pipeon-scheduled-execute)
  • Cancelar: chama POST /api/pipeon/scheduled/:id/cancel

/history — Historico

Exibe operacoes de pipeon_operations. Filtros: ticketId, procedureId, projectId, limit (maximo 500).

/restore — Restauracao de Backup

Carrega arquivo ZIP gerado pelo procedimento reset-evaluations e restaura os documentos na colecao evaluations do banco target-database.

07 Procedimentos Dinamicos

Procedimentos registrados em pipeon_procedures sao executados de forma generica pelo executor dinamico em /procedure/$id.

Como registrar um procedimento

  1. Acesse /admin > aba ProcedimentosDisponivel somente para role admin.
  2. Preencha os campos do procedimentoInforme nome, descricao, inputs necessarios e as operacoes MongoDB a executar nos steps.
  3. Revise o previewConfira nome, descricao, tabela de inputs e cards de steps.
  4. Selecione o projetoO procedimento herda o projectId do projeto escolhido.
  5. Clique em "Registrar procedimento"Salva em pipeon_procedures via POST /api/pipeon/procedures. Aparece automaticamente na lista de /procedure/.

Interpolacao de variaveis nos steps

Os valores dos inputs sao substituidos nos campos filter e update dos steps usando o padrao {{nomeDoInput}}.

Tipo do inputSubstituicaoExemplo
string Valor inserido diretamente como string "{{ticket}}" → "123456"
objectId "{{nome}}" (com aspas no JSON) → {"$oid":"valor"} "{{noticeId}}" → {"$oid":"65a1..."}

Exemplo de step com variaveis:

// Definicao no banco (pipeon_procedures.steps)
{
  "collection": "evaluations",
  "operation":  "updateMany",
  "filter":     { "notice": "{{noticeId}}" },
  "update":     { "$set": { "status": "NOTSTARTED" } }
}

// Apos interpolacao (noticeId = "65a1b2c3d4e5f6a7b8c9d0e1", type=objectId)
{
  "collection": "evaluations",
  "operation":  "updateMany",
  "filter":     { "notice": { "$oid": "65a1b2c3d4e5f6a7b8c9d0e1" } },
  "update":     { "$set": { "status": "NOTSTARTED" } }
}

08 Scheduler Automatico

O servidor Express executa um cron a cada minuto que verifica e executa procedimentos agendados automaticamente.

Disponibilidade O scheduler so funciona enquanto o processo local-api/server.js estiver rodando. Em producao (Cloudflare Worker), este recurso precisa de implementacao separada.

Fluxo de execucao automatica

  1. Busca de pendentesConsulta pipeon_scheduled por documentos com status: "pending" e scheduledFor <= agora.
  2. Backup automaticoBusca os documentos de evaluations no banco-alvo e salva um snapshot em pipeon_auto_backups (banco pipeon).
  3. Reset de statusExecuta updateMany com $set: { status: "NOTSTARTED" } e $unset dos campos de avaliacao.
  4. Limpeza de formularioExecuta updateMany com $unset dos campos phaseForm.
  5. LogRegistra resultado detalhado em pipeon_logs (banco pipeon).
  6. Marcacao como executadoSeta status: "executed" e executedAt: now em pipeon_scheduled.
  7. Recorrencia (se ativo)Se recurring: true, cria automaticamente o proximo agendamento para D+1 no mesmo horario recurringTime.

Tratamento de falhas

Em caso de erro em qualquer etapa, o documento em pipeon_scheduled recebe:

{
  status:    "failed",
  failedAt:  "2026-06-03T14:00:00.000Z",
  failError: "mensagem de erro"
}

O cron continua processando os proximos procedimentos — uma falha nao interrompe os demais.

10 Regras Criticas

Restricoes arquiteturais que nao devem ser violadas sob nenhuma circunstancia.

Banco alvo dos procedimentos Procedimentos no Pipeon so podem operar contra o banco target-database. Nunca apontar logica de procedimento para outros bancos. A pagina /procedure/ valida essa restricao antes de permitir qualquer execucao.
Separacao de bancos Dados de metadados do sistema (usuarios, logs, operacoes, agendamentos, backups, procedimentos) devem ser gravados EXCLUSIVAMENTE no banco pipeon. Nunca misturar com dados operacionais.
routeTree.gen.ts O arquivo src/routeTree.gen.ts e gerado automaticamente pelo TanStack Router. Nao edite manualmente — qualquer alteracao sera sobrescrita pelo servidor de desenvolvimento.
Autenticacao JWT em dev Em desenvolvimento, o JWT e assinado com o secret fixo "pipeon-local-dev-secret". Em producao, substitua por uma variavel de ambiente segura.
