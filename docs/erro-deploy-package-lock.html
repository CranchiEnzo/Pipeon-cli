Erro de deploy: package-lock.json fora de sincronia

O deploy para Cloudflare Workers falhou no passo npm ci porque o lock file não continha as dependências do Azure AD. Diagnóstico, correção aplicada e a decisão de fundo sobre bun versus npm.

01 O que aconteceu

Ao fazer merge para a main, o GitHub Actions disparou o deploy e quebrou no passo de instalação de dependências.

O workflow de deploy roda o comando npm ci antes de buildar e publicar no Cloudflare Workers. Esse passo terminou com exit code 1 e a seguinte mensagem:

# npm error code EUSAGE
npm error `npm ci` can only install packages when your package.json
and package-lock.json or npm-shrinkwrap.json are in sync.
Please update your lock file with `npm install` before continuing.

npm error Missing: @azure/msal-browser@5.14.0 from lock file
npm error Missing: @azure/msal-react@5.4.5   from lock file
npm error Missing: @azure/msal-common@16.9.0 from lock file
Sintoma O npm ci exige que package.json e package-lock.json estejam 100% sincronizados. Três pacotes Azure estavam declarados no package.json, mas ausentes do package-lock.json — então o passo abortou antes mesmo do build.

Vale notar a diferença entre os dois comandos de instalação do npm:

ComandoComportamento
npm installResolve dependências e conserta o lock file se estiver desatualizado.
npm ciInstalação estrita para CI: recusa rodar se o lock estiver fora de sincronia. Não conserta nada.

02 Por que aconteceu (causa raiz)

O projeto mantém dois arquivos de lock diferentes, atualizados por ferramentas diferentes — e eles saíram de sincronia.

No commit que preparou o ambiente para login Azure AD, as dependências foram adicionadas ao package.json:

// package.json
"@azure/msal-browser": "^5.14.0",
"@azure/msal-react":   "^5.4.5",

Porém o package-lock.json não foi atualizado junto. O motivo é estrutural: localmente o projeto usa bun como gerenciador de pacotes (que mantém o bun.lock), enquanto o CI usa npm (que lê o package-lock.json). O bun não toca no lock do npm.

Causa raiz Dois locks divergentes: o bun.lock (atualizado localmente pelo bun) e o package-lock.json (consumido pelo npm ci no CI). Mexer em dependências via bun atualiza só um deles, deixando o outro defasado.
AmbienteFerramentaLock consumido
Local (desenvolvimento)bunbun.lock
CI (deploy GitHub Actions)npm cipackage-lock.json

03 A correção aplicada

Regenerar o package-lock.json para incluir os pacotes Azure, sem reinstalar tudo.

npm install --package-lock-only

A flag --package-lock-only atualiza apenas o arquivo de lock, sem baixar/reinstalar pacotes no node_modules.

Executado e verificado Rodei o comando acima e confirmei que o package-lock.json passou a conter as referências @azure/msal (10 ocorrências) e que o arquivo ficou marcado como modificado no git status. A mensagem "up to date" do npm apareceu apenas porque o node_modules local já estava satisfeito (instalado via bun) — mas o lock em si foi atualizado, que é o que o CI precisa.
Ainda pendente A correção existe apenas na máquina local. É necessário commitar e fazer push do package-lock.json atualizado para destravar o deploy. Aguardando commit

04 Para que o bun é usado

Neste projeto, o bun é apenas uma ferramenta de desenvolvimento — não faz parte do código que roda em produção.

O bun aparece em dois lugares: na documentação (CLAUDE.md) e no script de inicialização local start-dev.ps1, cujo último comando é bun dev. E o ponto decisivo é que os scripts do package.json não usam nada exclusivo do bun:

"dev":    "vite dev",
"build":  "vite build",
"lint":   "eslint .",
"format": "prettier --write ."

Ou seja, bun dev apenas chama o vite por baixo. O bun não está atuando como runtime (o "motor" que executa o código, papel que aqui é do Node) — não há Bun.serve, bun:sqlite, nem imports de APIs do bun em lugar nenhum. Ele cumpre só dois papéis:

  • Instalar dependências (gerando o bun.lock).
  • Disparar os scripts, que no fim são vite, eslint e prettier — todas ferramentas Node padrão.
Em resumo O bun é usado como um "npm mais rápido" — gerenciador de pacotes e atalho para comandos. Nada do código depende dele para executar.

05 Migrar para npm é viável?

Sim, sem perda de funcionalidade. Como o bun não é runtime nem é importado pelo código, é uma tradução direta de comandos.

Hoje (bun)Equivalente (npm)
bun installnpm install
bun devnpm run dev
bun run buildnpm run build
bun run lintnpm run lint
bun run formatnpm run format

O backend local-api/ (Express) já roda em npm/Node. Padronizar em npm, portanto, unifica o projeto numa ferramenta só.

Manter os dois (bun local + npm no CI)

Risco recorrente

  • Dois locks que saem de sincronia.
  • O erro de deploy volta a cada mudança de dependência.
  • Exige disciplina manual de atualizar os dois.

Padronizar em npm

Resolve a causa raiz

  • Um único lock (package-lock.json).
  • Local e CI usam a mesma ferramenta.
  • O bug não volta a acontecer.
Único trade-off Perde-se velocidade de instalação: o bun instala pacotes mais rápido que o npm. Não há perda de funcionalidade nem de compatibilidade.

Passos para migrar de fato

  1. Ajustar o script de devTrocar bun dev por npm run dev no start-dev.ps1.
  2. Atualizar a documentaçãoTrocar os comandos bun por npm na seção de comandos do CLAUDE.md.
  3. Remover o lock do bunApagar o bun.lock para não restar fonte de divergência.
  4. Garantir o lock do npm em diaManter o package-lock.json sincronizado (já feito nesta correção).

07 Perguntas frequentes

Por que o npm disse "up to date" se faltavam pacotes?

Porque o node_modules local já tinha os pacotes (foram instalados via bun anteriormente). O --package-lock-only olha o estado instalado e regenera o lock a partir dele — então não precisou baixar nada, mas o package-lock.json foi de fato reescrito incluindo as dependências Azure.

Posso usar npm ci localmente para validar antes do push?

Sim, mas atenção: o npm ci apaga e recria o node_modules do zero a partir do lock. É exatamente o que o CI faz, então é o teste mais fiel. Só não rode se estiver no meio de um trabalho que dependa do estado atual do node_modules.

Se eu continuar usando bun, como evito que o erro volte?

Depois de qualquer alteração de dependência, rode também npm install --package-lock-only e commite o package-lock.json junto. É a correção pontual. A solução definitiva é padronizar em uma ferramenta só.

Migrar para npm quebra alguma coisa do projeto?

Não. O bun não é runtime aqui e não é importado pelo código — os scripts apenas chamam vite, eslint e prettier, que rodam igualmente sob Node/npm. O local-api já usa npm. A migração é uma troca de comandos, não de comportamento.
