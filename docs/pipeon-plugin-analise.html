01 Visao geral

O plugin integra o Claude Code ao fluxo de suporte interno [COMPANY], automatizando a triagem e a analise tecnica de chamados abertos pelo usuario final no [INTERNAL-SYSTEM].

Hoje, quando um operador recebe um chamado, ele precisa manualmente interpretar o relato, reproduzir o erro, consultar o codigo e montar uma resposta. Esse fluxo e lento e depende do conhecimento individual de cada operador sobre o sistema.

Com o plugin, o operador cola o texto do chamado e os prints de evidencia diretamente no Claude Code. O sistema le o monorepo [INTERNAL-SYSTEM], classifica automaticamente o tipo de problema e entrega um relatorio estruturado com a analise tecnica, o caminho para simular o erro (quando aplicavel) e a sugestao de acao — seja uma correcao de codigo, um ajuste direto no banco via CLI, ou uma instrucao didatica para repassar ao usuario final.

Principio central O Claude Code atua como um analista de primeiro nivel: ele le o contexto do sistema (via CLAUDE.md), vasculha o monorepo e entrega um diagnostico — cabendo ao operador validar e executar a acao recomendada.

O que o plugin nao faz

  • Nao aplica correcoes de codigo automaticamente em producao.
  • Nao executa procedimentos no banco sem confirmacao humana.
  • Nao responde diretamente ao usuario final — o operador sempre e o intermediario.

02 Classificacao da requisicao

O sistema recebe os inputs do chamado e determina automaticamente em qual das tres categorias o problema se enquadra antes de executar qualquer analise.

Inputs aceitos

  • Texto do chamado (obrigatorio) — descricao do problema conforme aberta pelo usuario final.
  • Prints / evidencias (opcional) — capturas de tela do erro, mensagens exibidas na interface, logs do console do navegador.
Qualidade do input Quanto mais detalhado o texto do chamado e mais claros os prints, mais precisa sera a classificacao. Chamados vagos como "nao funciona" podem gerar classificacoes ambiguas — nesse caso o plugin solicita informacoes adicionais ao operador antes de prosseguir.

Categorias de classificacao

Tipo Gatilho tipico Acao do sistema Output esperado
BUG Erro de tela, excecao no console, comportamento inesperado do sistema apos uma acao do usuario Le o monorepo, localiza o trecho de codigo responsavel, mapeia o caminho de reproducao Relatorio tecnico com arquivo, linha, stack e sugestao de fix
Ajuste em Base Dado incorreto no sistema, registro duplicado, campo com valor errado que precisa ser corrigido diretamente no MongoDB Identifica a colecao envolvida, verifica se existe procedimento no CLI, indica ou cria o procedimento adequado Nome do procedimento existente OU draft de novo procedimento pronto para revisao
Orientacao ao Usuario Duvida de uso, dificuldade de navegacao, recurso que o usuario nao sabe como acessar ou opera de forma incorreta Le a documentacao do monorepo, mapeia o fluxo de navegacao, identifica a rota e permissao envolvidas Instrucao didatica numerada para o operador testar em homologacao e repassar ao usuario

Nos casos em que o chamado apresenta caracteristicas de mais de uma categoria — por exemplo, um dado errado que tambem causa um erro de tela — o plugin identifica a classificacao primaria (a que bloqueia o usuario) e menciona a secundaria no relatorio como ponto de atencao adicional.

03 Fluxo: BUG

Acionado quando o sistema identifica um problema tecnico no codigo — erro de execucao, comportamento incorreto ou falha de integracao entre frontend e API.

Passo a passo

  1. Leitura do monorepo [INTERNAL-SYSTEM] O Claude Code usa as ferramentas Grep, Read e Glob para vasculhar o repositorio. O CLAUDE.md do [INTERNAL-SYSTEM] e lido primeiro para entender a estrutura do projeto antes de qualquer busca.
  2. Localizacao do trecho responsavel Com base no relato e nos prints, o sistema identifica os arquivos e funcoes envolvidos. No contexto do CLI, busca preferencialmente em src/routes/, src/lib/atlas.ts e src/lib/permissions.ts.
  3. Mapeamento do caminho de reproducao O sistema descreve, passo a passo, como um desenvolvedor pode reproduzir o erro localmente — incluindo pre-requisitos de dados no banco, rota a acessar, cargo necessario e acao exata a executar.
  4. Geracao da sugestao de correcao Com o contexto do erro localizado, o Claude Code elabora uma sugestao de fix: o arquivo a editar, a mudanca sugerida com trecho de codigo e o raciocinio tecnico por tras da correcao.

Estrutura do relatorio gerado

## Relatorio de BUG — [titulo-do-chamado]

Arquivo afetado:    src/routes/ciclos.tsx
Linha aproximada:   142
Funcao/componente:  CiclosPage

### Stack de chamadas
CiclosPage → atlasCall("aggregate", "ciclos", ...) → POST /api/mongodb/aggregate

### Reproducao local
1. Acesse /ciclos com cargo "operator" e permissao "ciclos" ativa
2. Clique em "Exportar relatorio" com periodo anterior a 01/01/2024
3. Observe o erro 500 retornado pela API

### Sugestao de correcao
// Antes
const pipeline = [{ $match: { data: { $gte: dateFrom } } }]

// Depois — garantir que dateFrom seja um objeto Date, nao uma string
const pipeline = [{ $match: { data: { $gte: new Date(dateFrom) } } }]
Validacao obrigatoria A sugestao de correcao e um ponto de partida tecnico, nao uma solucao definitiva. O desenvolvedor responsavel deve revisar o contexto completo, testar em homologacao e avaliar impactos colaterais antes de aplicar em producao.

04 Fluxo: Ajuste em Base

Acionado quando o problema nao esta no codigo, mas em dados incorretos no MongoDB — registros com valores errados, duplicidades, campos ausentes ou estados inconsistentes que precisam ser corrigidos diretamente no banco.

Passo a passo

  1. Identificacao da colecao envolvida O sistema analisa o relato para determinar qual colecao MongoDB esta impactada. Considera as colecoes conhecidas do [INTERNAL-SYSTEM] mapeadas no CLAUDE.md e as operacoes disponiveis via atlasCall() em src/lib/atlas.ts.
  2. Verificacao de procedimento existente no CLI O sistema consulta a lista de procedimentos cadastrados na rota /procedure do CLI para verificar se ja existe um processo adequado para o tipo de correcao identificada.
  3. Cenario A — Procedimento existente O sistema indica qual procedimento usar, descreve como acessa-lo na interface do CLI e orienta o operador sobre os parametros necessarios para executar a correcao com seguranca.
  4. Cenario B — Procedimento inexistente O sistema gera um draft de novo procedimento com todos os campos necessarios para que a equipe de produto/desenvolvimento possa revisar, testar e publicar como um novo processo reutilizavel no CLI.

Draft de novo procedimento (Cenario B)

## Draft de Novo Procedimento — [nome-sugerido]

Nome:         "Correcao de status de inscricao duplicada"
Descricao:    "Remove registros duplicados na colecao inscricoes gerados
               por falha de idempotencia no endpoint de submissao."

Colecao alvo:  inscricoes
Banco:         target-db  // sempre via override database: "target-db"
Operacao:      deleteMany  // via atlasCall()

Filtro sugerido:
{
  candidatoId: "<ID_DO_CANDIDATO>",
  cicloId:    "<ID_DO_CICLO>",
  createdAt:   { $gt: "<DATA_DO_INCIDENTE>" }
}

Criterio de aplicacao:
"Usar somente quando confirmado via find() que existem N > 1
 documentos com o mesmo candidatoId + cicloId para o mesmo periodo."

## Proximos passos para publicacao
1. Revisao tecnica pelo desenvolvedor responsavel
2. Teste em ambiente de homologacao com dados reais anonimizados
3. Aprovacao pelo gestor de produto
4. Cadastro no CLI via /admin → Procedimentos
Operacoes destrutivas exigem aprovacao Qualquer procedimento que envolva deleteMany, updateMany ou replaceOne em larga escala deve passar por revisao e aprovacao formal antes de ser executado. O draft gerado pelo plugin nunca e aplicado automaticamente.

05 Fluxo: Orientacao ao Usuario Final

Acionado quando o problema nao e um bug nem um dado errado, mas uma dificuldade do usuario em operar o sistema — seja por desconhecimento do fluxo, limitacao de permissao nao compreendida ou navegacao incorreta.

Passo a passo

  1. Leitura das documentacoes do monorepo O sistema le os arquivos de documentacao do [INTERNAL-SYSTEM] e o CLAUDE.md do CLI para entender os fluxos disponiveis e os requisitos de cada funcionalidade.
  2. Mapeamento do fluxo de navegacao Com base no relato, o sistema identifica qual rota o usuario estava tentando acessar ou qual acao estava tentando executar. Considera as rotas do CLI (/ciclos, /procedure, /restore, etc.) e os requisitos de permissao de cada uma.
  3. Identificacao do ponto de dificuldade O sistema determina onde a dificuldade ocorreu: ausencia de permissao para a funcionalidade, etapa do fluxo que o usuario nao completou corretamente, ou comportamento esperado do sistema que o usuario interpretou como erro.
  4. Geracao da instrucao didatica O sistema produz um passo a passo claro e numerado para o operador seguir em homologacao antes de repassar ao usuario. Inclui a verificacao de permissao necessaria via hook usePermission() e a rota exata envolvida.
  5. Script de resposta ao usuario Alem da instrucao tecnica interna, o sistema gera um texto pronto para o operador adaptar e enviar ao usuario final — em linguagem simples, sem jargao tecnico.

Estrutura do relatorio gerado

## Orientacao ao Usuario — [titulo-do-chamado]

Rota envolvida:       /ciclos
Permissao necessaria:  usePermission("ciclos:edit") → true
Cargo minimo:         operator com permissao ciclos:edit ativa (ou admin)

### Instrucao para o operador (testar em homologacao)
1. Acesse /admin → Permissoes e confirme que o cargo do usuario
   tem "ciclos:edit" habilitado
2. Faca login com uma conta de teste com esse cargo em homologacao
3. Navegue ate /ciclos — o botao "Editar" deve aparecer no canto superior direito
4. Confirme que o botao esta visivelmente ativo (nao desabilitado)
5. Clique em Editar, altere um campo e salve — confirme o toast de sucesso

### Script de resposta ao usuario final
"Para editar um ciclo, seu perfil precisa ter a permissao de edicao
 habilitada pelo administrador do sistema. Caso voce nao visualize o
 botao 'Editar', entre em contato com o administrador para que ele
 verifique as permissoes do seu cargo em Configuracoes > Permissoes."
Homologacao obrigatoria O operador deve sempre testar o fluxo em ambiente de homologacao antes de repassar a instrucao ao usuario final. Isso garante que a orientacao e valida para a versao atual do sistema e evita passar informacoes incorretas.

07 Roadmap de implementacao

Etapas ordenadas para colocar o plugin em operacao — da configuracao do contexto ao primeiro uso real com chamados do [INTERNAL-SYSTEM].

  1. Definir estrutura do CLAUDE.md no monorepo [INTERNAL-SYSTEM] Documentar rotas, colecoes MongoDB, endpoints da API, permissoes, fluxos criticos e convencoes do codigo — esse arquivo e o "mapa" que o plugin vai ler primeiro em cada analise.
  2. Criar a skill /analisar-chamado Adicionar o arquivo .claude/commands/analisar-chamado.md no repositorio CLI (ou no [INTERNAL-SYSTEM]). O arquivo define o prompt de sistema, as instrucoes de classificacao e o formato esperado para cada tipo de relatorio.
  3. Implementar a logica de classificacao Escrever as instrucoes de prompt que ensinam o Claude Code a distinguir entre BUG, Ajuste em Base e Orientacao — com exemplos de cada tipo para calibrar a classificacao.
  4. Integrar leitura do monorepo via tools nativos Validar que o Claude Code consegue usar Grep, Read e Glob no repositorio [INTERNAL-SYSTEM] a partir do contexto da skill. Mapear os caminhos criticos que o plugin deve priorizar em cada tipo de analise.
  5. Implementar gerador de novo procedimento CLI Para o fluxo de Ajuste em Base sem procedimento existente: criar o template de draft e integrar com o modelo de dados da rota /procedure do CLI para que o draft gerado seja diretamente importavel.
  6. Criar pasta docs/analises/ e definir convencao de nomes Estrutura de arquivo: AAAA-MM-DD-slug-do-chamado.md. Considerar se os relatorios devem ser commitados no repositorio ou mantidos apenas localmente pelo operador.
  7. Testes com chamados reais em homologacao Selecionar 5 a 10 chamados historicos do [INTERNAL-SYSTEM] (um de cada tipo) e executar o plugin. Avaliar precisao da classificacao, qualidade do relatorio e tempo de execucao. Iterar no prompt da skill com base nos resultados.
  8. Documentar processo de uso para operadores [COMPANY] Criar um guia rapido (pode ser um segundo HTML com tema [COMPANY]) explicando como instalar o Claude Code, como usar o comando /analisar-chamado e como interpretar os tres tipos de relatorio gerados.

Perguntas em aberto

O plugin precisa de acesso ao repositorio [INTERNAL-SYSTEM] em tempo real?

Sim. O Claude Code precisa ter o repositorio [INTERNAL-SYSTEM] clonado localmente (ou acesso via MCP server de filesystem) para usar as ferramentas Grep, Read e Glob. Uma alternativa e exportar snapshots do codigo relevante para um diretorio acessivel ao plugin — mas isso adiciona complexidade de manutencao.

Como lidar com chamados que nao se encaixam em nenhuma categoria?

O plugin deve reconhecer quando nao tem contexto suficiente para classificar com confianca e, nesses casos, retornar uma lista de perguntas estruturadas para o operador coletar mais informacoes antes de uma segunda analise. Nunca deve forcar uma classificacao incorreta.

Os relatorios gerados devem ser commitados no repositorio?

Recomendavel adicionar docs/analises/ ao .gitignore inicialmente. Os relatorios contem dados de chamados que podem ser sensiveis. Uma vez que o processo estiver maduro, avaliar se faz sentido versionar os relatorios anonimizados como base de conhecimento para melhorar o prompt da skill ao longo do tempo.

Qual o modelo Claude recomendado para a skill?

Para analise tecnica de qualidade — especialmente no fluxo de BUG, que exige leitura e raciocinio sobre codigo — recomenda-se usar Claude Opus 4.8 ou Sonnet 4.6. O modelo pode ser especificado no cabecalho da skill com o parametro model. Para chamados simples de orientacao, Haiku 4.5 pode ser suficiente e mais economico.
