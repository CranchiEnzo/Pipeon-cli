Pipeon Local API

Servidor Express rodando em http://localhost:5000 — intermediário entre o frontend e o MongoDB.

🗄 Banco Pipeon

Dados internos do sistema: usuários, logs, agendamentos, backups e configurações. Configurável via PIPEON_MONGO_URL.

🎯 Banco dos Sistemas

Dados de avaliação (target-database etc.). Connection string definida pela sessão selecionada no login.

🔐 Autenticação JWT

Todos os endpoints (exceto /api/auth/login) exigem token Bearer no header Authorization.

⏱ Scheduler

Cron interno verifica a cada minuto procedimentos com scheduledFor ≤ agora e os executa automaticamente.

Autenticação

Após o login, inclua o token em todas as requisições:

# Header obrigatório em todos os endpoints (exceto login)
Authorization: Bearer eyJhbGciOiJIUzI1NiIsInR5cCI6IkpXVCJ9...

O token expira em 8 horas. Ao expirar, o frontend redireciona automaticamente para o login.

Separação de bancos

# Toda requisição ao Pipeon vai para um de dois caminhos:

Ação do usuário no app
        │
        ├── Dados de avaliação (evaluations, notices...)
        │       └── MongoClient(connectionString da sessão)
        │               └── db("target-database")
        │
        └── Dados internos do Pipeon (logs, agendamentos...)
                └── MongoClient(PIPEON_MONGO_URL)
                        └── db("pipeon")
Importante: os endpoints /api/pipeon/* sempre usam o banco Pipeon, independente da sessão ativa do usuário. Nenhum dado interno é gravado em target-database.

Startup & Configuração

O servidor lê variáveis do arquivo local-api/.env ao iniciar:

# local-api/.env

# Connection string do MongoDB dedicado ao Pipeon
PIPEON_MONGO_URL=mongodb://localhost:27017/

# Nome do banco dentro do servidor acima (padrão: pipeon)
# PIPEON_DB_NAME=pipeon

Saída esperada ao iniciar com npm start:

Pipeon Local API rodando em http://localhost:5000
Banco Pipeon  : mongodb://localhost:27017/pipeon
Banco padrão  : mongodb://localhost:27017/
Scheduler     : verificando procedimentos a cada minuto

Auth

POST /api/auth/login Autenticar usuário e obter token JWT público ▶
🗄 pipeon · pipeon_users

Autentica o usuário pelo e-mail. A senha não é validada neste modo — o acesso depende apenas do cadastro no banco Pipeon. Retorna um JWT válido por 8 horas.

Body (JSON)
emailobrig.E-mail cadastrado em pipeon_users
// Request
POST /api/auth/login
{
  "email": "usuario@empresa.com"
}

// Response 200
{
  "token": "eyJhbGci..."
}

// Response 401 — e-mail não encontrado
{ "message": "Usuario nao encontrado no banco local." }

POST /api/users Criar novo usuário 🔐 auth ▶
🗄 pipeon · pipeon_users
Body (JSON)
nameobrig.Nome completo do usuário
emailobrig.E-mail único de login
roleobrig.admin | operator | user
passwordopc.Não validado atualmente

{ "id": "507f1f77bcf86cd799439011" }
POST /api/users/:id/update Atualizar dados de um usuário 🔐 auth ▶
🗄 pipeon · pipeon_users

Aceita qualquer campo no body — aplica $set nos campos enviados.

// Exemplo: desativar usuário
POST /api/users/507f1f77bcf86cd799439011/update
{ "isActive": false }

// Response 200
{ "success": true }

Agendamentos

Gerenciamento dos procedimentos agendados. O scheduler verifica pipeon_scheduled a cada minuto e executa automaticamente procedimentos vencidos.

GET /api/pipeon/scheduled Listar todos os agendamentos 🔐 auth ▶
🗄 pipeon · pipeon_scheduled

Retorna até 200 documentos ordenados por scheduledFor decrescente.

// Campos do documento retornado
{
  "_id": "507f...",
  "procedureName": "Retornar avaliações para pendentes...",
  "ticket": "123456",
  "noticeId": "65a1b2c3d4e5f6a7b8c9d0e1",
  "evaluatorUserId": "65a1...",
  "evaluatorName": "João Silva",
  "evaluatorEmail": "joao@empresa.com",
  "scheduledFor": "2026-06-10T14:00:00.000Z",
  "scheduledBy": "admin@empresa.com",
  "database": "target-database",
  "connectionString": "mongodb://...",   // null se local
  "recurring": false,
  "recurringTime": null,
  "status": "pending",   // pending | executed | cancelled | failed
  "createdAt": "2026-06-02T10:00:00.000Z",
  "executedAt": null,
  "autoExecuted": false
}
GET /api/pipeon/scheduled/due Listar procedimentos pendentes vencidos 🔐 auth ▶
🗄 pipeon · pipeon_scheduled

Retorna documentos com status = "pending" e scheduledFor ≤ agora. Usado pelo frontend para exibir o banner de execução no menu.

POST /api/pipeon/scheduled Criar novo agendamento 🔐 auth ▶
🗄 pipeon · pipeon_scheduled
// Response 200
{ "insertedId": "507f1f77bcf86cd799439011" }
POST /api/pipeon/scheduled/:id/cancel Cancelar um agendamento 🔐 auth ▶
🗄 pipeon · pipeon_scheduled

Define status = "cancelled". O scheduler não executa procedimentos cancelados.

{ "success": true }

Define status = "executed" e preenche executedAt. Chamado pelo frontend após execução manual de um procedimento agendado.

Backups

O servidor busca as avaliações diretamente no banco-alvo e salva no banco Pipeon, sem tráfego de documentos via HTTP.

POST /api/pipeon/backup/create Gerar e salvar backup server-side 🔐 auth ▶
🗄 pipeon · pipeon_auto_backups
O servidor conecta simultaneamente ao banco-alvo (para ler as avaliações) e ao banco Pipeon (para salvar o backup). Chamado durante execuções manuais e automáticas (cron).
Body (JSON)
noticeIdobrig.ObjectId do ciclo (string hex 24 chars)
evaluatorUserIdobrig.ObjectId do avaliador
ticketopc.Número do chamado
evaluatorNameopc.Nome do avaliador
evaluatorEmailopc.E-mail do avaliador
databaseopc.Nome do banco-alvo (padrão: target-database)
connectionStringopc.Connection string do banco-alvo (padrão: localhost)
// Response 200
{
  "backupId": "507f1f77bcf86cd799439011",
  "documentCount": 42
}

// Documento salvo em pipeon_auto_backups
{
  "ticket": "123456",
  "noticeId": "65a1...",
  "evaluatorName": "João Silva",
  "database": "target-database",
  "documentCount": 42,
  "executionType": "manual",  // "manual" | "automatic"
  "createdAt": "2026-06-02T14:30:00.000Z",
  "documents": [...]   // array completo das avaliações
}
Como encontrar backups no MongoDB:
use pipeon
db.pipeon_auto_backups.find({ ticket: "123456" })
db.pipeon_auto_backups.find({ evaluatorEmail: "joao@..." })

Proxy MongoDB genérico

Endpoint flexível que roteia operações MongoDB para qualquer banco e collection. Usado pelo frontend para todas as operações nos dados de avaliação.

POST /api/mongodb/:action Executar operação MongoDB 🔐 auth ▶
Atenção: este endpoint conecta ao banco definido no body, usando a connection string da sessão ativa. Não deve ser usado para coleções do Pipeon — use os endpoints /api/pipeon/* para isso.
Actions disponíveis em :action
ActionEquivalente MongoDB
findcollection.find(filter)
findOnecollection.findOne(filter)
insertOnecollection.insertOne(document)
updateOnecollection.updateOne(filter, update)
updateManycollection.updateMany(filter, update)
replaceOnecollection.replaceOne(filter, replacement)
aggregatecollection.aggregate(pipeline)
Body (JSON) — campos comuns
databaseobrig.Nome do banco (ex: target-database)
collectionobrig.Nome da collection (ex: evaluations)
connectionStringopc.MongoDB URI (padrão: localhost:27017)
filteropc.Filtro MongoDB. Suporta {"$oid":"..."} e {"$date":"..."}
pipelineopc.Pipeline de agregação (apenas action aggregate)
// Exemplo — buscar avaliações de um ciclo
POST /api/mongodb/find
{
  "database": "target-database",
  "collection": "evaluations",
  "filter": {
    "notice": { "$oid": "65a1b2c3d4e5f6a7b8c9d0e1" }
  },
  "limit": 100
}

// Response 200
{ "documents": [...] }

VariávelPadrãoDescrição
